Implementation · 6 stages

Ship a vertical slice
at the end of every stage.

69 designs is a menu, not a build order. Below is the sequence that minimises rework, lets real jumpers test the app within 2–3 weeks of starting, and keeps each stage independently shippable. Hand each stage to Claude Code as a single scoped job.

Vertical slices, not horizontal

Each stage ships a working UI + backend + auth slice. Never "build all the UIs, then wire up the data."

Real jumpers in stage 2

By the end of stage 2 (≈ week 3) you should be able to put TestFlight builds in front of 5–10 friendly skydivers and let them log jumps.

Admin gates everything

Stage 1's admin shell only needs price + feature flags + manual user creation. Build it minimal — flesh out in stage 5.

FOUNDATION Stage 01 ≈ 1 week

Foundation — auth, schema, billing

Nothing else works without this. Get Supabase Postgres + auth + Stripe webhooks working end-to-end with a single seed user. No real UI needed beyond a "you're logged in" placeholder and the most basic admin pages.

Build
  • DBSchema: users, sessions, subscriptions, audit_log, app_settings (price lives here)
  • APISupabase auth: email/password, JWT, password reset, magic-link option
  • $$Stripe: webhook for invoice.paid / failed / refunded; subscription enforcement middleware
  • RBACRoles: user, support, finance, admin, super-admin (just super-admin needed live)
  • UIMobile: Splash, Welcome, Sign in, Create, Forgot, Reset, Paywall (screens 01–06)
  • UIWeb: Sign up, Payment, Success, Log in, Forgot, Reset (screens 08–13)
  • UIAdmin: Login + bare Admin Settings (screen 15) for setting the price
Skip (for now)
  • —Push notifications · QR sign-off · gear / certs · stats · offline sync
  • —All marketing pages except a minimal homepage placeholder
  • —Admin discounts, flagged entries, announcements — comes later
Reality check
  • !Use Stripe Sandbox + test cards. Don't take real money yet.
  • !Price & trial length come from app_settings table — never hardcode $5
Stage 1 is done when
  • A new user can sign up via web, get a 30-day trial, pay $5 via Stripe test card, and the subscription status reflects in Postgres within 5 sec.
  • That same user can sign in on iOS / Android and see a "you're logged in" stub.
  • A super-admin can sign in, change the price from $5 to any other value, and the next signup pays the new price.
  • Audit log records every price change, signup, payment, password reset.
CORE LOOP Stage 02 ≈ 2 weeks

The mobile core loop — log a jump

This is the entire value of the product. By the end of this stage, a real skydiver can install the app, log a jump in under a minute, see it in their list, view its detail, and sync it across devices. Nothing else matters as much as this.

Build
  • DBSchema: jumps (24 cols), tags, jump_tags, signatures, dropzones, aircraft
  • APICRUD jumps + soft delete + edit history; tag autocomplete; DZ / aircraft search
  • UIJump list (screen 07A — default; B / C available behind Settings toggle)
  • UIFilter & sort bottom sheet (screen 08)
  • UICreate jump 1 / 2 / 3 / signature (screens 10–13) + saved confirmation (14)
  • UIJump detail A (15) + Edit (15)
  • UIFavourites (16), Browse by tag (17), Manage tags (34)
  • UIProfile (29) + Edit profile (30) — minimum needed to show "who you are"
Skip (for now)
  • —QR sign-off (18) + instructor scan (19) — defer to stage 6
  • —Jump detail B / C and Stats overview B / C variants — wait for user vote
  • —Offline sync — defer to stage 6; for now assume always-online
Reality check
  • !TestFlight + internal Play Store track. Recruit 5–10 friendly jumpers from your home DZ now.
  • !Instrument log_jump events — you want to know how long it actually takes.
Stage 2 is done when
  • A jumper can log a complete jump (signature included) in < 60 seconds on a phone.
  • The same jump appears on their list within 2 seconds, on a second device within 10.
  • 5+ external testers have logged at least one real jump and given feedback.
CORE LOOP Stage 03 ≈ 1.5 weeks

Stats, gear, certificates

Round out the mobile app. None of these are blockers for "can I log a jump" but they turn the app from a logger into a logbook companion. Currency & repack alerts are the single most-requested feature from licensed jumpers.

Build
  • DBSchema: gear, gear_service_log, certificates, currency_window
  • APIMaterialised views for stats (avoid re-computing on every request)
  • UIStats overview A (20), jumps over time (21), type pie (22), aircraft/DZ (23)
  • UIGear list (24), Gear detail (25), Add gear (26)
  • UICertificates list (27), Add cert (28)
  • UISettings (31) — needed for units toggle, display-layout picker, currency thresholds
  • UISubscription management on mobile (32)
Skip (for now)
  • —Notification settings (35) — wait for stage 6 push infra
  • —Document attachment for certs (PDFs) — out of scope until stage 6
Reality check
  • !Currency rules vary by jurisdiction (APF / USPA / BPA / CSPA differ). Make thresholds admin-configurable.
  • !Gear service intervals are configurable too — Cypres 2 = 4yr, M2 = "lifetime," etc.
Stage 3 is done when
  • Stats reflect every logged jump within 10 seconds of save.
  • A user with overdue gear sees a red badge on the gear list within 24h of due date passing.
  • Cert expiry stages (30d / 7d / day-of) trigger the right UI state.
PLATFORM Stage 04 ≈ 1 week

Public website + web logbook

Now you have something worth marketing. Build the public site, the read-only web logbook (for desktop review and instructor verification), and the basic account area.

Build
  • UIMarketing pages 01–07 (Homepage, Features, Pricing, About, Contact, Privacy, Terms)
  • APILive stats endpoint for homepage counter (cached 5 min)
  • UIAccount dashboard (14), Logbook viewer (15) — read-only, search + filter + pagination
  • UIAccount settings (16), Web subscription (17), Billing history (18), Delete account (19)
  • APIAccount deletion: soft-delete + 30-day grace window + final purge cron
Skip (for now)
  • —Blog / changelog / press kit — nice-to-haves, not in the 22-page brief
Reality check
  • !Homepage stats need a job that aggregates platform totals nightly — don't query live tables.
  • !Open SEO + Plausible analytics. App Store / Play Store deep links work from the homepage.
PLATFORM Stage 05 ≈ 1.5 weeks

The full admin panel

You've been operating with stage-1's minimal admin. Now build it out so you can actually support users, respond to billing issues, run promos, and watch the business.

Build
  • UIDashboard (01) with KPIs + signups chart + live activity
  • UIUsers (02), User detail (03), User jump log (04)
  • UIRevenue (05), Subscriptions (06), Manual override (07)
  • UIPlans & pricing editor (08), Discounts (09), Apply discount (09b)
  • UIPlatform stats (10), Flagged entries (11), Data export (12)
  • UISupport tickets (13), Announcements (14), Admin settings (15)
  • APIFlagging algorithm: implausible FF, exit-alt > ceiling, future-dated, duplicate #
Reality check
  • !Every admin action logs to audit_log with admin_id, reason, timestamp — non-negotiable.
  • !Discount + price changes flow through the same app_settings + coupons tables that stage 1 stubbed.
  • !Announcements need a push provider — sets you up for stage 6 if not already.
FINISH Stage 06 ≈ 1.5 weeks

Finishing touches

The features that turn this from "works" into "the one jumpers actually use." Save these for last — each one is meaningful but none block launch. Ship them in any order based on user feedback.

Build
  • PDFExport logbook (33): APF / USPA / CSPA / BPA layouts, CSV, share sheet
  • SYNCOffline-first: local SQLite, sync queue with conflict resolution
  • PUSHFCM / APNs: currency, repack, cert expiry, instructor sign-off requests
  • QRQR sign-off (18) + instructor scan (19) — signed JWT, 5-min expiry
  • UINotification settings (35), Hero-screen variants (B/C), Jump-detail variants (B/C), Stats variants (B/C)
Reality check
  • !Offline sync is hard. Budget 2× your estimate. Test on actual DZ wifi.
  • !QR sign-off is the only marketing-differentiating feature — ship it well or don't ship it.
  • !The 6 hero-screen variants are user-pickable in Settings → Display layout, per the design.
All six stages done when
  • You can stop the dev work and the app keeps running for 30 days without your intervention.

How to hand each stage to Claude Code

Open this roadmap on a stage. Then run the Handoff to Claude Code flow from the project menu, scoped to that stage only — for example: 

Build stage 02 of the SkydiveLog roadmap. Scope:
  • Mobile screens 07, 08, 10–17, 29–30 from Mobile Designs.html
  • Postgres schema: jumps, tags, jump_tags, signatures, dropzones, aircraft
  • Supabase RLS so users can only see their own data
  • Don't touch QR sign-off, stats, gear, or offline sync — those are stage 3 / 6
Acceptance: a TestFlight build a real jumper can log a jump in < 60 seconds.

Then review the resulting PR per stage. Don't move on to the next stage until the current one passes its acceptance criteria, has a deploy on a real environment, and ideally has at least one user smoke-testing it.
